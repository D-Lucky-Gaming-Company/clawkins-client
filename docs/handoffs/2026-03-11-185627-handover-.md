## Current TestGame Changes (Applied)

### 1) Map collision and borders
- Movement is now map-aware via `MoveSystem.setMap(...)`.
- Player bounds are clamped to real `.tmx` dimensions (map edge), not fixed world constants.
- Collision is read from tile collision objects instead of generic full-tile blocking:
  - Uses tile-object rectangles (`RectangleMapObject`) for overlap checks.
  - Uses a feet-focused player hitbox for more natural pass-behind behavior.
- Collision layers are restricted to tile layers **before** the `objects` layer:
  - Foreground/overlay layers after `objects` are ignored for blocking.
  - This fixes the top-side "1 tile early collision" issue from visual-only layers.

### 2) Input/movement speed behavior
- Default movement is now faster ("sprint by default").
- Holding `Shift` slows movement.
- Implemented by:
  - adding `baseSpeed` to `Move` component,
  - applying speed multipliers in `PlayerInputSystem`.

### 3) System wiring updates
- `GameScreen` now passes map changes to `MoveSystem`, in addition to render/camera map consumers.
- `PlayerInputSystem` now focuses on input + animation state only.

### Key observations relevant to encounter and turn-based design
- Encounter trigger is collision-driven in world exploration:
  - `PlayerPhysicsComponent` sends `START_BATTLE` on foe collision.
  - `GameScreen` receives it and switches to `BattleScreen`.
- Battle system is event/observer based:
  - `BattleState` extends `BattleSubject`.
  - UI/HUD observes battle events and updates accordingly.
- Turn flow is stateful and timed:
  - Player/opponent phases are sequenced via battle events and timers.
  - Combat resolution is centralized in `BattleState`.
- Data-driven enemy/player stats:
  - Uses entity config properties for HP/AP/DP/SPD and reward values.
- Separation of concerns:
  - World map logic is separate from battle screen logic.
  - Transition in/out of battle keeps world state continuity.

### What to reuse conceptually in TestGame
- Trigger encounters from world collision/events.
- Keep battle logic in a dedicated state/service layer.
- Keep battle UI separate from battle rules/turn resolution.
- Use event notifications so map, battle, and UI systems stay decoupled.

## Boilerplate Plan: Encounter + Turn-Based System for TestGame

This is a minimal scaffold, aligned to current ECS + map structure.

### A) Encounter trigger (world side)

#### New components
- `EncounterZone`
  - fields: `encounterId`, `encounterTableId`, `isOneShot`
- `EnemyParty`
  - fields: list of enemy definitions for fixed encounters
- `EncounterTrigger`
  - tag on entities that can start combat

#### New systems
- `EncounterDetectionSystem`
  - family: player + transform and encounter entities + transform
  - when overlap occurs: dispatch `StartEncounterEvent`
- `EncounterTransitionSystem`
  - listens for `StartEncounterEvent`
  - freezes world input/update
  - pushes `BattleScreen` (or battle state mode)

#### Data source
- Option 1 (fastest): use Tiled object names/properties in `objects` layer.
- Option 2 (scalable): `encounters.json` with encounter tables and weights.

### B) Turn-based core (battle side)

#### New domain objects
- `BattleContext`
  - player snapshot, enemy party, rng seed, encounter metadata
- `BattleUnit`
  - hp/mp/ap/dp/spd/status, alive state
- `TurnOrder`
  - queue/list sorted by initiative/speed

#### New services
- `BattleStateMachine`
  - states:
    - `INIT`
    - `PLAYER_COMMAND`
    - `PLAYER_RESOLVE`
    - `ENEMY_COMMAND`
    - `ENEMY_RESOLVE`
    - `TURN_END`
    - `VICTORY`
    - `DEFEAT`
    - `ESCAPE`
- `BattleResolver`
  - resolves action -> damage/heal/status outcome
- `BattleEventBus` (or simple observer)
  - emits events for UI and rewards

#### New actions
- `ATTACK`
- `SKILL` (placeholder)
- `ITEM`
- `DEFEND`
- `ESCAPE`

### C) Battle UI boilerplate

#### New UI classes
- `BattleCommandPanel`
  - emits player action selection
- `BattleStatusPanel`
  - shows HP/MP/turn order
- `BattleLogPanel`
  - shows combat text

#### UI behavior
- Only enabled during `PLAYER_COMMAND`.
- Disabled during resolve phases.
- Reacts to battle events (`damage`, `status`, `turn start/end`, `victory/defeat`).

### D) Integrating with current TestGame architecture

#### Integration points
- `TiledObjectConfigurator`
  - spawn encounter entities from map object names (e.g. `Enemy`, `EncounterZone`)
- `GameScreen`
  - register `EncounterDetectionSystem`
  - register a transition handler that switches from exploration loop to battle loop
- `AssetService`
  - add battle data assets (enemy defs, encounter tables)

#### Save/resume rules (initial)
- On battle start: snapshot player combat-relevant state.
- On battle win: apply rewards/xp/items then return to map.
- On escape: return to map, optional penalty/cooldown.
- On defeat: game over or respawn policy.
